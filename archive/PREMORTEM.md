## 범위 전제

- 도메인: 점찍어둔 아파트 한 채에 대한 다관점 검증 (Second Opinion)
- 인터페이스: Streamlit 텍스트 기반
- 에이전트: MC, 시세·입지·리스크·재무·미래가치 분석가, 서기(종합 리포터)
- MVP: 한 매물에 대한 7-scene 플로우 5분 데모 (인터뷰 → 매물 주소 → 5인 분석 → 종합 리포트)

## 시나리오 1: 5명이 다 비슷하게 말한다 (검증 시스템이 추천 시스템이 됐다)

### 상상된 실패

> "5명 분석가가 전부 '괜찮은 매물입니다, 추천드려요'라고 말했다. 별점도 다 4점대였다. 사용자는 '그래서 사라는 거야?'라며 5명이 있는 의미가 없었다. Second Opinion 컨셉이 뭉개졌다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| 검증·반박 강제 규칙 | 각 분석가 프롬프트에 "반드시 1개 이상의 의심·반박·리스크"를 강제 | Phase 1 |
| 별점 흩어짐 검증 | 5인 응답에서 별점 표준편차가 너무 낮으면(<0.5) 자동 챌린지 | Phase 4 |
| 영역 분리 테스트 | 같은 매물에 5인 응답 나란히 놓고 영역 침범·중복 점검 | Phase 1·6 |
| Gold Standard 갱신 | "별점이 흩어진 케이스" 시연 시나리오 demo_mock에 박기 | Phase 6 |

### 상상된 실패

> "시세 분석가가 'P50 6.2억'이라고 했는데 출처가 빠졌다. 사용자가 '진짜야?'라고 물으니 답을 못 했다. 검증 시스템인데 검증의 핵심 데이터에 신뢰가 무너졌다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| MOLIT API 조기 검증 | 국토부 API 키 발급 + 단지·평형 P50 호출 검증 | Phase 1 |
| 출처 강제 가드 | 시세·재무·미래가치 분석가 응답에 출처 누락 감지 → warning | Phase 1 (기존 source_validator 재활용) |
| 표본 부족 처리 | 동일 평형 거래 N<5면 "표본 부족, 신뢰구간 넓음" 명시 + 별점 신중 | Phase 1 |
| 헤도닉 보정 한계 명시 | 층·향·리모델링 미반영을 항상 캡션에 표기 | Phase 1 |

### 상상된 실패

> "리스크 분석가가 '이 매물은 좋은 입지에 있어서 사도 됩니다'라고 입지·추천을 했다. 미래가치 분석가가 '월 원리금이 만만치 않습니다'라고 재무를 말했다. 5명이 있는데 사실상 1명이 5번 말하는 것과 다르지 않았다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| 영역 경계 명시 | 각 분석가 프롬프트에 "발언 금지 영역" 목록 + 위반 시 자동 필터 | Phase 1 |
| 영역 침범 테스트 | "DSR 계산해줘"를 입지 분석가에게 → "재무 분석가 영역" 답변 확인 | Phase 7 |
| 대조 테스트 | 같은 매물에 5인 응답 나란히 놓고 영역 표시 색깔로 구분 | Phase 6 |
| 페르소나 명세서 갱신 루프 | 영역 침범 발견 → 즉시 agents/*.md 수정 → 재테스트 | 상시 |

## 시나리오 4: 매물 주소 변환 실패 (Scene 03 막힘)

### 상상된 실패

> "사용자가 '광명시 OO아파트 105동 1203호'를 입력했는데 Kakao API 호출에서 좌표 변환이 실패했다. 다음 단계로 못 넘어가서 5명 분석이 시작도 안 됐다. UX 첫 막힘이 가장 치명적인 막힘이 됐다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| Kakao API 조기 검증 | Phase 3 시작 전에 API 키 발급 + 한국 주소 패턴 다양성 테스트 | Phase 3 |
| 자유 입력 → 후보 N개 표시 | 1개로 단정 못 하면 후보 3개 표시 → 사용자 선택 | Phase 3 |
| 좌표 변환 실패 fallback | API 실패 시 사용자에게 "아파트명·동·호수만 입력" 폼 제공 | Phase 3 |
| 지도 표시 실패 fallback | 지도 안 떠도 텍스트 컨펌만으로도 다음 단계 진행 가능 | Phase 3 |

## 시나리오 5: 종합 리포트가 한 화면에 안 들어간다 (Scene 06 실패)

### 상상된 실패

> "5명이 각자 길게 말했는데 서기가 종합을 못 했다. 별점은 5개인데 합의 결론이 'AI는 결정을 대신할 수 없습니다'만 반복. '5명 중 3명이 신중 검토 권합니다' 같은 진짜 합의가 안 나왔다. 결국 사용자가 길을 잃었다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| 별점 산출 규칙 명시 | 각 분석가 응답에서 별점 1~5점을 구조화 출력으로 강제 (JSON) | Phase 4 |
| 합의 결론 알고리즘 | 별점 평균 + 표준편차 → "추천/긍정 신호/신중 검토/주의 필요" 규칙 정의 | Phase 4 |
| 핵심 쟁점 추출 | 점수 가장 높은 카테고리 vs 가장 낮은 카테고리를 한 줄로 ("X vs Y") | Phase 4 |
| 한 화면 검증 | 종합 리포트 1080p에서 스크롤 없이 다 보이는지 측정 | Phase 6 |

## 시나리오 6: 7-scene 플로우가 end-to-end로 안 된다

### 상상된 실패

> "MC 인터뷰는 작동, 매물 주소 입력은 작동, 5명 분석은 작동, 종합 리포트도 작동. 그런데 Stage 간 데이터 전달이 망가져서 종합 리포트에 매물 주소가 안 떴다. 부분 완성, 전체 미완성."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| 파이프라인 먼저 | 개별 분석가 완성도보다 7-scene 흐름 연결 우선 | Phase 1·2·3 |
| 단위 테스트 (E2E) | BuyerProfile + 매물 주소 → 5인 분석 → 종합 리포트 통합 테스트 | Phase 1·6 |

## 시나리오 7: "ChatGPT랑 뭐가 달라요?" (검증 컨셉이 안 와닿는다)

### 상상된 실패

> "발표 후 'ChatGPT에 5명 역할 나눠서 물어보면 되지 않나요?'라는 질문에 막혔다. Second Opinion 차별점을 30초 안에 설명 못 했다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| 비교 시연 | 같은 매물을 ChatGPT에 물어본 응답 vs 우리 시스템 5인 별점 종합 리포트 나란히 | Phase 6 |
| 차별점 3줄 | ① 국토부 P50 직접 연동 + 출처 자동 ② 5인 영역 분리 + 검증·반박 강제 ③ 한 화면 별점 종합 + 합의 결론 | Phase 6 |
| 검증 vs 자문 vs 추천 | "우리는 검증 시스템이라 '사세요/사지 마세요'를 안 합니다. 별점·합의로 판단 근거를 드리고, 결정은 사용자가 하십니다" — 발표 스크립트 명시 | Phase 6 |

### 상상된 실패

> "미래가치 분석가가 '역세권 신설 호재 있음 → 5점'을 줬다. 그런데 그 호재는 이미 무산된 계획이었다. 출처도 5년 전 신문 기사. 검증해야 할 호재가 검증되지 않은 채 별점에 반영돼 시스템 신뢰도가 무너졌다."

| 대응 | 구체적 행동 | 시기 |
|---|---|---|
| 호재 출처 의무 | 모든 호재에 "출처 + 발표일 + 실현 일정" 필수. 단순 기대치 금지 | Phase 1 |

## 핵심 교훈

### 1. "검증은 추천이 아니다"

5명이 다 좋다고 하면 검증 시스템이 아니다. **별점이 흩어지는 것이 정상**이고, 시스템이 만들어야 할 가치다. "신중 검토" 결론도 명시적 결정으로 인정한다.

### 2. "출처가 곧 신뢰다"

시세 분석가가 말하는 P50, 재무 분석가가 말하는 LTV·DSR, 미래가치 분석가가 말하는 호재 — 모두 출처가 없으면 ChatGPT와 똑같다. 출처는 우리의 핵심 차별점이다.

### 3. "한 화면이 곧 결정 가능성이다"

5명이 길게 말해도 사용자가 종합 못 하면 결정 못 한다. **종합 리포트 1페이지에 별점·합의·핵심 쟁점·후속 액션**이 한 번에 보여야 한다. 한 화면 안에 결정의 모든 재료가 있어야 한다.

### 4. "결정 책임은 사용자에게 돌려준다"

"AI는 결정을 대신하지 않는다. 사용자를 더 똑똑하게 만들어준다." 강매·강요·확정 권유 금지. 마지막 결정은 항상 사용자.
