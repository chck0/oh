# 핵심 기술 결정 — BADUGI

> Why Tree(docs/whytree.md)의 요약 + 에이전트가 빠르게 참조할 수 있는 결정 로그

---

## 결정 1: 직장 주소를 검색 시작점으로

**결론**: 검색의 축은 집이 아닌 직장.
**이유**: 사람들이 아파트를 고를 때 첫 질문은 항상 "거기서 출퇴근 가능해?"이기 때문.

---

## 결정 2: 통근버킷 × 평형 매트릭스

**결론**: (버킷, 평형) 슬롯당 최저가 1개 추천. 같은 단지 중복 제거.
**이유**: "가장 좋은 집"이 아니라 "내 조건에서 가장 합리적인 집"을 찾아야 하기 때문.
버킷 구간: 10분 단위 (0~30, 30~40, 40~50, 50~60분).

---

## 결정 3: ODsay 멀티키 병렬 호출

**결론**: ODsay API를 최대 20개 키로 병렬 호출, DB 셀 단위 캐시.
**이유**: 카카오 대중교통 API는 상업적 규모에서 비용·rate limit 제약이 크고, ODsay는 셀 단위 병렬화가 용이.
**제약**: Vercel 60초 → `MAX_FETCH_CELLS=200`, 초과분 `deferred` 처리.

---

## 결정 4: LLM 역할 분리

**결론**: 추천 카드 → `claude-sonnet-4-6` (2~3문장), 일반 카드 → `claude-haiku-4-5` (한 줄 40자).
**이유**: 룰 기반 수치 계산 + LLM 자연어 표현 분리로 할루시네이션 최소화. 비용 최적화.

---

## 결정 5: SQLite ↔ Supabase 이중 어댑터

**결론**: `DATABASE_URL` 유무로 자동 전환. 방언 차이는 `app/portable.py` 한 곳에서 처리.
**이유**: 로컬 개발에서 Postgres 설치 진입 장벽 제거. 코드 분기 없이 두 환경 지원.
**주의**: pgBouncer Transaction mode → `prepared_statement=False` 필수.

---

## 결정 6: Vanilla JS (프레임워크 없음)

**결론**: HTML + Vanilla JS + CSS. `web/` 정적 파일을 Vercel CDN 직접 서빙.
**이유**: MVP 단계에서 번들러·빌드 파이프라인은 오버헤드. Kakao Maps SDK가 전역 스크립트 방식.
**전환 기준**: 팀 확장 또는 컴포넌트 재사용 필요성이 생기면 React 도입 재검토.

---

## 결정 7: 친구 채팅 — 인라인 패널

**결론**: 단지별 컨텍스트(실거래·POI)를 자동 주입한 채팅 패널을 화면 오른쪽 슬라이드인으로 구현.
**이유**: 외부 채팅 앱으로 우회하면 DB 컨텍스트가 사라짐. 인라인이면 `apt_seq` 기준으로 시세·도보 데이터 바로 주입 가능.
**구현 범위**: spec-22(기본) → spec-23(고도화) → spec-24(웹검색 tool_use) → spec-27(파일 첨부).

---

## 결정 8: 이미지·PDF만 첨부 지원 (Word/PPT 제외)

**결론**: 이미지(Claude Vision) + PDF(pypdf 텍스트 추출)만 지원.
**이유**: python-docx/pptx → lxml(C 확장) 의존 → Vercel Python 런타임 빌드 실패.
부동산 맥락에서 분양 사진·계약서 PDF가 90% 이상의 실사용 케이스.

---

## 결정 9: localStorage 상태 관리 — 서버 저장 없음

**결론**: favorites(♥), 검색 히스토리, 채팅 대화를 모두 localStorage에 저장. 서버 DB 미사용.
**이유**: 로그인 없는 서비스에서 서버 사용자 식별이 불가. 프라이버시 민감 데이터(예산·주소)를 서버에 남기지 않는 것이 신뢰 포인트.
**구현**: favorites(spec-08), search history(spec-15), chat history 24시간(spec-23).
**제약**: 기기 간 동기화 없음. 브라우저 캐시 삭제 시 소실.

---

## 결정 10: 가격 변동률 — 최근 3개월 vs 이전 6개월 비교

**결론**: `price_chg_6m_pct` = (최근 3개월 평균 − 이전 6개월 평균) / 이전 6개월 평균.
**이유**: "최근"과 "과거"를 명확히 분리해야 트렌드가 의미 있음. 직전 거래 1개 비교는 노이즈가 크다.
**표시**: +5% 이상 → 🔴 상승, −5% 이하 → 🟢 하락, 그 외 → 표시 없음 (spec-16).

---

## 결정 11: 직접 검색 단지 — `_isLookup` 임시 삽입 트릭

**결론**: 직접 검색한 단지를 `resultData.cards`에 `_isLookup:true` 플래그와 함께 임시 삽입.
**이유**: `showDetail(aptSeq)`가 지도 이동 시 `resultData.cards`에서 lat/lng를 조회. 직접 검색 단지는 cards에 없어서 지도 이동 불가. `showDetail` 내부 수정 시 기존 flow 깨짐 위험.
**구현**: `_openLookupDetail()`에서 삽입 → `showDetail()` 재사용 (spec-26).

---

## 결정 12: 통근 경제성 요금 구간 공식 (spec-28)

**결론**: 수도권 교통카드 편도 요금을 편도 시간으로 근사.
```
≤25분 → 1,500원   (단거리 직통)
≤40분 → 1,800원   (중거리 or 환승 1회)
≤55분 → 2,100원   (장거리 or 환승 2회)
60분+ → 2,300원
월 교통비 = 요금 × 2(왕복) × 20(영업일)
연 통근시간 = 편도시간 × 2 × 20 × 12 ÷ 60 (시간)
```
**버킷 내 최단 대비 diff**: `build_recommendations()`에서 버킷별 min 계산 후 `annual_hours_diff` 필드 추가.

## 결정 13: 친구 채팅 컨텍스트 주입 구조

**결론**: 채팅 열 때 `apt_seq` 기준으로 시세·실거래·POI·통근시간을 자동 주입. 사용자가 별도 설명 없이도 단지 맥락 질문 가능.
**진화**: spec-22(기본 Q&A) → spec-23(인메모리 캐시+대화 유지) → spec-24(tool_use 웹검색, 최대 3턴) → spec-25(스마트 Chip) → spec-26(직접 검색 연동) → spec-27(이미지·PDF 첨부).
**웹검색**: DuckDuckGo Instant Answer API (무료, API 키 불필요). 실패 시 graceful degradation.
**모델**: Claude Opus (친구 채팅 메인). 캐시 TTL 1시간, history_len ≤ 2.
