# 프로젝트 저널 — BADUGI가 만들어진 과정

> 실제 대화·인터뷰에서 나온 결정들의 기록.
> "왜 이렇게 됐는가"를 이해하려는 에이전트와 팀원을 위한 문서.

---

## 1단계: 출발점 — "왜 직장 주소인가"

**초기 가설**: "아파트를 찾는 사람에게 가장 중요한 것은 가격이다."

**대화에서 발견한 반례**:
> "가격은 이미 네이버부동산에서 다 볼 수 있잖아요. 근데 저는 거기서 봐도 어디를 봐야 할지 모르겠더라고요."

→ 정보는 넘치는데, **시작점이 없다**는 것이 진짜 문제.
→ 출발점을 "집"이 아닌 "직장"으로 바꾸면 검색 범위가 즉시 정해진다.

**결정**: 직장 주소 → 대중교통 통근시간 → 반경 내 매물 필터 (spec-01, spec-02)

---

## 2단계: 첫 번째 피벗 — "최저가 1개"의 역설

**초기 설계**: 통근시간 버킷별로 가장 저렴한 1개를 추천하면 사용자가 바로 결정할 수 있다.

**5인 인터뷰에서 깨진 가설 (2026-05)**:

5인 전원이 같은 질문을 했다: **"왜 이게 싸요?"**

실제 발언록:
> "1층이라서 싸다, 탑층이다 등 데이터가 카드에 같이 적혀있어야 신뢰하고 볼 수 있을 것 같다."

> "1억 싸게 나온 매물은 보통 이유가 있다. 외벽 누수, 북향, 위층 소음…"

**핵심 인사이트**: 최저가 = 매력이 아니라 **의심 신호**.
"왜 싼지" 설명 없이는 신뢰를 얻지 못한다.

**결정**: Why-price 태그 의무 노출 → spec-11 구현 (이유 칩 3개)

---

## 3단계: 타겟 포지셔닝 — "누구를 위한 서비스인가"

**인터뷰 결과 정리 (2026-05)**:

| 인터뷰어 | 적합도 | 발언 요약 |
|---------|--------|-----------|
| 30대 1인 가구 첫 구매 | ✅ 완전 적합 | "딱 내가 원하던 것" |
| 갈아타기 고수 | ❌ 부적합 | "학군·재건축 없으면 반쪽짜리" |
| 맞벌이 | ⚠️ 부분 | "남편 직장만 넣으면 의미 없어요" |
| 투자 관심 | ❌ 부적합 | "GTX 호재를 모르는 데이터는 못 믿겠다" |
| 1인 가구 여성 | ⚠️ 부분 | "야간 귀가 안전도 통근시간만큼 중요해요" |

**대화에서 나온 결론**:
> "타겟을 넓히면 뭘 잘하는 서비스인지 모호해진다. 좁게 가자."

→ **"부동산을 한두 번 알아본 30대 1인 가구 첫 집 구매자"** 로 타겟 확정.
→ 맞벌이는 별도 기능(spec-13)으로 대응하되, 핵심 타겟은 유지.

## 4단계: 기능 확장의 흐름 — 대화에서 결정된 순서

### "비교가 판단을 만든다" → 즐겨찾기·비교 기능

대화:
> "좋아 보이는 거 저장해두고 나중에 비교하고 싶은데, 지금은 그게 안 되네요."

→ spec-08 (즐겨찾기 ♥) → spec-14 (최대 3개 나란히 비교)

### "두 직장이 있는 가정은 어떻게?"

대화:
> "저는 남편이랑 직장이 달라요. 한 명 직장만 기준으로 하면 다른 사람이 희생하는 거잖아요."

→ spec-13 (Dual Workplace) — 업계 최초 두 직장 교집합 기능

### "카드에서 더 깊이 물어보고 싶을 때"

대화:
> "이 단지 재건축 얘기 있는 것 같던데, 확인하고 싶어요."
> "분양 팸플릿 PDF가 있는데 이거 분석해줄 수 있어요?"

→ spec-22~27 친구 채팅 진화: 기본 Q&A → 웹검색 → 파일 첨부

### "월 얼마나 더 쓰는 건지 알고 싶다"

대화 (현재 세션):
> "월교통비 60천원, 연 통근 시간은 어떻게 계산 된건지??"

→ 사용자가 수치 의미를 궁금해한다 = 계산 근거가 UI에 보여야 함을 재확인
→ spec-28 (통근 경제성) — 계산 공식을 명시적으로 설계

## 5단계: 실패에서 배운 것 (git 히스토리 기반)

### 실패 1: onclick 인라인 핸들러 + 한국어 아파트명
커밋 `0fce62f`: `fix(spec-22): 친구 채팅 onclick — data 속성 방식으로 변경`

> 아파트명에 특수문자(괄호, 따옴표)가 들어가면 onclick="..." 안의 JS가 파싱 오류를 냄.
> → data-* 속성 + addEventListener로 전면 교체.

### 실패 2: Vercel에서 lxml 빌드 실패
커밋 `b419805`: `fix(spec-27): Vercel 빌드 오류 수정 — python-docx/pptx 제거`

> Word/PPT 지원을 위해 python-docx를 추가했더니 Vercel Python 런타임에서 빌드 실패.
> lxml이 C 확장 라이브러리라 Vercel이 컴파일 못 함.
> → 이미지·PDF만 지원, Word/PPT는 "PDF로 변환 후 첨부" 안내.

### 실패 3: 직접 검색 단지 지도 이동 불가
커밋 `7b4abfc`: `fix(spec-26): 직접 검색 카드 — 추천 카드 동일 구조`

> 직접 검색한 단지는 resultData.cards에 없어서 showDetail() 호출 시 지도가 안 움직임.
> → cards에 _isLookup:true 플래그로 임시 삽입해서 기존 함수 재사용.

### 실패 4: 가격 범위 넓게 설정 → 504 타임아웃
> max_price를 넓게 잡으면 ODsay 호출 셀 수가 폭증해서 Vercel 60초를 초과.
> → MAX_FETCH_CELLS 상한 + 범위 경고 UI 추가.
