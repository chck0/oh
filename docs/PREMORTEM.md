# Data 기반 Multi-Agent 생애 첫 주택 구매 자문 시스템 Pre-mortem

> KAIST IMMS MBA 맥락 + 내 집 마련 에이전트 페르소나 반영
> "학기 말, 이 프로젝트가 망했다고 가정하자. 왜 망했을까?"

---

## 범위 전제

* 도메인: 생애 첫 주택 구매 자문 (무주택 실수요자)
* 인터페이스: 텍스트 채팅 (음성 삭제)
* 에이전트: MC 인터뷰어, 중개사, 재무설계사, 시장분석가, 비서실장
* MVP: "마포구 첫 주택 구매 검토" 5분 데모 (3단계 플로우)

---

## 시나리오 1: 실거래 데이터를 못 가져온다

### 상상된 실패

> "국토교통부 MOLIT API를 연동하려 했는데 파싱에서 막혔다. 시장분석가 에이전트가 'P50은 7.8억'이라고 말했지만 출처가 없었다. '출처 있는 숫자만 말한다'는 핵심 가치가 1번인데, 정작 출처를 댈 데이터가 없었다. 호갱노노 크롤러도 차단당해 완전히 맹탕이 됐다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|-|-|-|
| API 조기 테스트 | 국토부 getRTMSDataSvcAptTrade API 키 발급 + 테스트 호출 | 1주차 |
| 크롤러 fallback | CrawlerBlockedError 발생 시 MOLIT API 데이터로 graceful fallback | 2주차 |
| 최소 데이터 기준 | 마포구/공덕 아파트 최근 3개월치만 있으면 MVP 가능 | 3주차 판단 |
| 정적 데이터 허용 | API + 크롤러 모두 실패 시 샘플 데이터로 대체, 사용자에게 명시 | 상시 |

---

## 시나리오 2: 재무설계사가 근거 없는 숫자를 말한다

### 상상된 실패

> "재무설계사 에이전트가 'DSR 40% 기준 최대 대출 4.2억'이라고 했다. 출처를 물었더니 계산 근거 없이 LLM이 지어낸 거였다. 시연 중에 이 응답이 나왔다. 실제 소득과 DSR 규제를 반영하지 않은 숫자가 첫 주택 구매자를 잘못 이끌 수 있다는 게 현실이 됐다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|-|-|-|
| 프롬프트 방어 | "BuyerProfile의 소득·자기자본 없이는 대출 한도를 말하지 마세요" 명시 | 3주차 |
| 출처 강제 포맷 | 재무설계사 응답에 계산 근거 필수. 근거 없으면 수치 삭제 | 4주차 |
| 할루시네이션 테스트 | BuyerProfile 없는 상태로 질문해서 "먼저 소득을 알려주세요" 답변 확인 | 6주차 |
| 시연 사전 검증 | 데모 시나리오의 모든 수치와 DSR 공식 수동 대조 | 발표 2주 전 |

---

## 시나리오 3: 세 전문가가 역할을 침범한다

### 상상된 실패

> "재무설계사가 '마포구보다 성동구가 더 괜찮아요'라며 동네 추천을 했고(중개사 영역), 중개사가 'DSR 40%면 4억까지 빌릴 수 있어요'라며 대출 계산을 했고(재무설계사 영역), 시장분석가가 '이 가격이면 괜찮은 투자예요'라며 매입 판단을 내렸다(중개사·재무설계사 영역). 셋 다 비슷한 말을 하니까 3명이지만 1명과 대화하는 느낌이었다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|-|-|-|
| 영역 경계 명시 | 각 에이전트 프롬프트에 "발언 금지 영역" 목록 포함 | 3주차 |
| 영역 침범 테스트 | "DSR 계산해줘"를 중개사에게 물어서 "그건 재무설계사 영역"이라고 답하는지 확인 | 5주차 |
| 대조 테스트 | 같은 매물에 대해 3명 응답을 나란히 놓고 영역 구분 확인 | 6주차 |
| Gold Standard | "마포구 첫 주택 구매 검토" 시나리오를 3인 역할 버전으로 재작성하여 기준 설정 | 3주차 |
| 반복 튜닝 | 대조 테스트에서 겹치면 프롬프트 수정 → 재테스트 | 6주차~ |

## 시나리오 4: 시장분석가가 예스맨이 된다

### 상상된 실패

> "시장분석가가 하방 리스크를 제기해야 하는데, '네, 이 가격이면 적정합니다. 타이밍도 나쁘지 않아 보입니다'만 반복했다. 입주 물량 증가, 금리 리스크, 전세가율 하락 등 실제 구매자가 알아야 할 하방 시나리오를 한 번도 말하지 않았다. 결국 시장분석가가 있는 이유가 없었다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|-|-|-|
| 강제 하방 리스크 규칙 | 시장분석가 프롬프트에 "반드시 1개 이상의 하방 리스크를 지적하라. 동의만 하는 것은 금지" 명시 | 3주차 |
| 동의 감지 | 시장분석가 응답에 리스크 언급 없으면 자동으로 "입주 물량이나 금리 리스크는?" 후속 프롬프트 | 5주차 |
| 반론 품질 체크 | 시장분석가가 지적한 리스크가 실거래 데이터 기반인지 팀이 검증 | 7주차 |

## 시나리오 5: 3단계 플로우가 end-to-end로 안 된다

### 상상된 실패

> "MC 인터뷰어, 3인 에이전트 각각은 작동하는데, 전체 파이프라인이 연결 안 됐다. Stage 1(MC 인터뷰) → Stage 2(전문 자문) → Stage 3(호가 적정성) 흐름 중간에서 BuyerProfile이 전달이 안 됐다. COMPLETE_THRESHOLD=60점을 달성했는데도 Stage 2로 넘어가지 않았다."

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|-|-|-|
| 파이프라인 먼저 | 개별 에이전트 완성도보다 3단계 전체 흐름 연결 우선 | 4주차 |
| BuyerProfile 직렬화 | BuyerProfile JSON이 Stage 간에 정확히 전달되는지 단위 테스트 | 3주차 |
| 시나리오 고정 | "마포구 첫 주택 구매 검토" 딱 하나만 완벽하게 | 상시 |
| 주간 통합 테스트 | 매주 1회 전체 파이프라인 Stage 1~3 처음~끝 실행 | 5주차~ |

## 핵심 교훈

### 1\. "데이터가 곧 신뢰다"

시장분석가가 말하는 P50에 출처가 없으면 서비스 가치가 0이다. 국토부 출처 포맷 [LAWD_CD, N=건수, 조회일]이 신뢰의 근거다.

### 2\. "맥락 없이 자문 없다"

BuyerProfile 없이 시작하는 자문은 ChatGPT와 다를 바 없다. MC 인터뷰어가 먼저 상황을 파악하는 것이 차별점의 시작이다.

### 3\. "역할 경계가 곧 차별점이다"

중개사·재무설계사·시장분석가가 각자 자기 영역에서만 전문적으로 말하는 것이 "ChatGPT 3번 물어보기"와의 결정적 차이다. 이 경계가 무너지면 차별점도 무너진다.
