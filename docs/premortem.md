# Pre-mortem — BADUGI가 실패한다면

"6개월 후 BADUGI가 망했다. 왜였을까?"를 지금 미리 상상하는 문서.
각 시나리오마다 현재 대응 상태와 추가 조치를 정리한다.

---

## 기술 실패 시나리오

### T1. ODsay API가 막힌다
**시나리오**: ODsay 서비스 종료 또는 정책 변경으로 대중교통 데이터를 못 가져온다.
**현재 대응**: 멀티키(최대 20개)로 rate limit 분산, DB 캐시로 재호출 최소화.
**추가 조치 필요**: 카카오 대중교통 API를 fallback으로 준비. 캐시 히트율이 높아지면 신규 호출 비중 자체가 낮아짐.

### T2. Vercel 60초 타임아웃으로 첫 검색이 자꾸 실패한다
**시나리오**: 신규 직장 주소의 첫 검색에서 ODsay 호출이 60초를 초과해 504 에러.
**현재 대응**: `MAX_FETCH_CELLS=250` 상한으로 타임아웃 방지, `partial=true` 응답으로 재검색 유도.
**추가 조치 필요**: 백그라운드 워커(Supabase Edge Function 또는 별도 서버)로 캐시 프리워밍.

### T3. Supabase Free tier 500MB 초과
**시나리오**: trade_history 등 대용량 테이블이 Free 한도를 초과해 DB가 잠긴다.
**현재 대응**: trade_recent(3개월)만 기본 조회, trade_history는 상세 화면에서만 사용.
**추가 조치 필요**: 오래된 trade_history 주기적 정리 또는 Supabase Pro($25/월) 전환 기준 미리 설정.

### T4. Kakao Maps API 키 노출
**시나리오**: 프론트엔드에 하드코딩된 Kakao JS 앱키가 악용된다.
**현재 대응**: 등록된 도메인에서만 동작하도록 Kakao 콘솔에서 제한.
**추가 조치 필요**: 키 교체 절차 문서화, 이상 호출 모니터링.

## 사용자 경험 실패 시나리오

### U1. 결과 카드가 너무 많아서 선택을 못 한다
**시나리오**: 버킷 × 평형 슬롯이 많아지면 추천 카드가 10개 이상 나와 오히려 혼란.
**현재 대응**: 슬롯당 1개, 같은 단지 중복 제거.
**추가 조치 필요**: 최대 추천 카드 수 상한(예: 6개) 설정, 나머지는 "더 보기"로 접기.

### U2. AI 코멘트가 뻔하고 신뢰가 안 간다
**시나리오**: "좋은 입지입니다" 수준의 코멘트가 반복되어 사용자가 무시하게 된다.
**현재 대응**: 프롬프트에 장단점 균형, 구체적 수치 포함 지시.
**추가 조치 필요**: 코멘트 품질 주기적 샘플링, 프롬프트 버전 관리(A/B 테스트).

### U3. 모바일에서 지도 + 카드 동시 탐색이 불편하다
**시나리오**: 작은 화면에서 지도와 카드 리스트를 왔다갔다 하는 UX가 불편해 이탈.
**현재 대응**: 반응형 레이아웃 적용.
**추가 조치 필요**: 모바일 전용 UX 분리(지도 우선 / 리스트 우선 토글).

## 비즈니스·운영 실패 시나리오

### B1. 데이터가 오래되어 신뢰를 잃는다
**시나리오**: 실거래 데이터가 몇 달째 업데이트 안 되어 호가와 차이가 크다.
**현재 대응**: trade_recent는 최근 3개월 기준.
**추가 조치 필요**: 데이터 갱신 파이프라인 스케줄링 + 마지막 업데이트 날짜 UI 표시.

### B2. 공공데이터 API가 변경된다
**시나리오**: 국토부·한국부동산원 API 스펙 변경으로 파이프라인이 깨진다.
**현재 대응**: scripts/ 파이프라인 분리, 백엔드와 독립.
**추가 조치 필요**: API 버전 고정, 주요 엔드포인트 헬스체크 스크립트.

### B3. 경쟁 서비스가 같은 기능을 출시한다
**시나리오**: 직방·네이버 부동산이 통근시간 기반 검색을 도입한다.
**현재 대응**: "친구 같은 톤"과 AI 코멘트로 차별화.
**추가 조치 필요**: 빠른 반복으로 기능 선점. 코어 UX(5분 내 결정 보조)를 더 완성도 있게.
