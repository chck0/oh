# BADUGI 특허 기술 명세서 (Patent Technical Specification)

> **목적**: 특허 출원 청구항 도출 및 향후 구현 추가 시 갱신용 기술 트래킹 문서  
> **최초 작성**: 2026-05-28  
> **관리 원칙**: 새 spec이 추가될 때마다 "8. 향후 확장 기술" 섹션을 업데이트하고, 특허 출원 청구항에 반영 여부를 표시한다.

---

## 1. 발명의 명칭

**대중교통 통근시간 및 직장 위치 기반 부동산 자동 추천 시스템 및 방법**  
*(System and Method for Automatic Real Estate Recommendation Based on Public Transit Commute Time and Workplace Location)*

---

## 2. 기술 분야 요약

본 발명은 사용자의 직장 주소를 입력으로 받아, 대중교통 API를 통해 실측된 통근시간과 실거래 가격 데이터를 결합한 **부동산 자동 추천 엔진**에 관한 것이다. 특히 통근시간을 일정 구간(버킷)으로 분류하고, 평형 유형과의 교차 슬롯에서 최저가 매물을 자동 선정하는 매트릭스 추천 알고리즘과, 두 개의 직장 위치를 동시에 고려하는 교집합 필터링 방법을 핵심 기술로 한다.

---

## 3. 기존 기술의 한계 (배경기술)

| 기존 서비스 | 한계 |
|---|---|
| 네이버부동산, 호갱노노 | 주거 지역 중심 검색. 직장까지 실제 대중교통 시간 미제공 |
| 직방, 다방 | 지하철역 기반 거리. 실측 ODsay 경로 미적용 |
| 카카오맵 부동산 | 단순 지역 필터. 다중 조건 매트릭스 추천 없음 |
| 전체 | 맞벌이 두 직장 교집합 추천 기능 없음 (조사 기준 2026-05) |

## 4. 핵심 발명 구성 요소 (청구항 후보)

### 4-A. 통근버킷 × 평형 매트릭스 추천 알고리즘 ★ (독립항 1 후보)

**개념**  
대중교통 통근시간을 N분 단위 구간(버킷)으로 분류하고, 각 (버킷, 평형유형) 교차 슬롯에서 실거래 최저가 매물 1건을 자동 선정하는 방법.

**구성 단계**
1. 사용자가 직장 주소, 최대 통근시간(T_max), 관심 평형 유형(P_list), 최대 매매가를 입력한다.
2. 외부 지오코딩 API(Kakao REST)로 직장 주소를 위도·경도(lat, lng)로 변환한다.
3. 변환된 좌표로부터 반경 R_km = (T_max − 15) × 20 / 60 내의 아파트 단지를 필터링한다.
4. 대중교통 경로 API(ODsay)를 통해 각 단지까지의 실측 대중교통 총 소요시간(total_time_min)을 계산한다. API 호출 결과는 (직장ID, 단지그리드키) 쌍을 기준으로 DB에 캐싱하여 재호출을 방지한다.
5. total_time_min을 기준으로 버킷을 생성한다: [0, 30), [30, 40), [40, 50), [50, T_max) 등 10분 단위 슬라이스.
6. 실거래가 DB에서 사용자가 선택한 평형 유형 및 최대 매매가 조건을 만족하는 거래 내역을 조회한다.
7. 각 매물 카드에 (bucket_idx, pyeong_type) 슬롯을 부여한다.
8. 슬롯별로 실거래 최저가(price_low) 오름차순 정렬 후, 동일 단지 중복 없이 1건을 추천(is_recommended=True)으로 선정한다.
9. 추천 사유(pick_reason)를 자동 생성한다: "30~40분·30평대 중 최저가. 최단권보다 N천만 저렴."

**기술적 특징**
- 가격과 통근시간이라는 두 독립 축으로 구성된 2차원 매트릭스 구조
- 동일 단지 중복 추천 방지 로직 (같은 apt_seq는 첫 슬롯에만 추천)
- 결정론적 알고리즘 (동일 입력 → 동일 추천, 재현 가능)

### 4-B. 두 직장 교집합 기반 맞벌이 추천 ★ (독립항 2 후보)

**개념**  
두 개의 직장 주소를 입력받아, 두 직장 모두의 반경 내에 위치한 단지만 후보로 선정하고, `max(t1, t2)`를 기준으로 버킷 분류하는 방법.

**구성 단계**
1. 사용자가 직장 주소 #1(WP1) 및 직장 주소 #2(WP2)를 입력한다. WP2는 선택사항.
2. 두 직장 모두에 대해 지오코딩 후 wp_id를 발급한다. 두 주소가 동일한 경우 오류를 반환한다.
3. **교집합 필터링**: 두 직장 좌표로부터 각각의 반경 R_km 이내에 동시에 위치하는 단지만 후보로 선정한다. (논리곱 AND 조건)
4. **동적 API 예산 분배**: 대중교통 API 총 호출 한도(예: 200회)를 WP1, WP2의 캐시 미스 수 비율에 따라 동적으로 분배한다. (n1 미스, n2 미스 → 한도 내 균형 분배)
5. WP1→단지 통근시간(t1), WP2→단지 통근시간(t2)를 각각 병렬 조회한다.
6. 각 카드의 대표 통근시간 = `max(t1, t2)`으로 설정하여 버킷 분류 및 추천 매트릭스를 수행한다.
7. 추천 사유에 두 직장 맥락을 자동 포함한다: "두 직장 모두 30분 이내 · '20평대' 슬롯 최소가"

**기술적 특징**
- 단일 직장 모드와 완전 하위 호환 (WP2 미입력 시 기존 동작 동일)
- GREATEST()/MAX() DB 함수를 SQLite(개발)/Postgres(운영) 환경별 자동 전환
- 대중교통 API 예산의 동적 분배로 호출 수 최적화

### 4-C. LLM 코멘트 비동기 생성 및 (직장-단지-평형) 3키 캐싱 (종속항 후보)

**개념**  
추천 매물 카드에 AI 언어모델(LLM)로 생성한 맞춤형 코멘트를 API 응답 지연 없이 비동기 제공하고, (직장ID, 단지ID, 평형유형) 3개 키 조합으로 캐싱하는 방법.

**구성 단계**
1. 검색 API 응답 시 `llm_pending=true` 플래그를 함께 반환하고, LLM 코멘트 생성은 BackgroundTask로 분리한다.
2. 추천 카드(소수): 고성능 LLM(Claude Sonnet)으로 장단점 균형 2문장 코멘트 생성.
3. 일반 카드(다수): 경량 LLM(Claude Haiku)으로 핵심 1문장 코멘트 생성. Semaphore(8)로 동시 호출 수 제한.
4. 고성능 LLM 실패 시 경량 LLM으로 자동 폴백.
5. 생성 결과를 `(apt_seq, pyeong_type, wp_id)` 복합 키로 DB에 upsert 캐싱. 동일 조합 재검색 시 LLM 미호출.
6. 클라이언트는 `llm_pending=true` 수신 후 폴링 엔드포인트(`GET /api/comments`)로 완료 여부 확인.

### 4-D. 대중교통 API 멀티키 병렬 호출 및 그리드 셀 캐싱 (종속항 후보)

**개념**  
대중교통 경로 API를 다수의 인증 키로 동시 병렬 호출하고, 지리적 격자(그리드) 단위로 캐싱하여 중복 호출을 방지하는 방법.

**구성 단계**
1. 아파트 단지를 일정 크기의 지리적 그리드 셀(grid_key)로 분류한다. 같은 그리드 내 단지들은 동일 셀로 취급.
2. 직장 ID + 그리드 셀 키 조합으로 캐시 테이블(transit_cache) 조회. 캐시 히트 시 API 미호출.
3. 캐시 미스 셀에 대해 복수의 API 인증 키를 라운드로빈으로 순환하며 병렬 호출(최대 N키 × M 동시 = K 동시 호출).
4. 특정 인증 키 실패 시 다음 키로 자동 폴백. 호출 결과를 transit_cache에 저장.
5. Vercel 서버리스 60초 제약 내에서 처리할 수 있는 셀 수(250) 초과 시 `partial=true` 반환, 잔여는 다음 검색에서 점진적 누적.

---

### 4-E. 도보권 POI 카테고리 기반 지도 마커 연동 (종속항 후보)

**개념**  
단지 상세 화면에서 도보권 주변 시설 목록을 카테고리별로 분류 표시하고, 시설 클릭 시 지도 API를 통해 해당 시설의 실시간 좌표를 조회하여 인터랙티브 마커를 표시하는 방법.

**구성 단계**
1. DB의 도보 시설 데이터(poi_lclas_cd, poi_mlsfc_cd)를 카테고리 코드 기준으로 그룹화하여 표시한다.
2. 서브카테고리 허용 목록으로 오분류 데이터(주유소·아파트단지 등 L03~L06)를 필터링한다.
3. 사용자가 시설 항목을 클릭하면, 지도 SDK 내장 장소 검색 API로 시설명 + 아파트 중심점 반경 내 좌표를 실시간 조회한다.
4. 검색 1순위 결과로 카테고리별 색상 구분 마커를 지도에 표시하고, 클릭 시 InfoWindow(시설명, 도보 거리·시간)를 표시한다.
5. 다른 시설 클릭 시 이전 마커를 제거하고 새 마커로 교체(중첩 방지). 같은 항목 재클릭 시 토글 제거.

## 5. 시스템 구성도 (논리 흐름)

```
[사용자 입력]
  직장주소(WP1) [, 직장주소(WP2)]
  최대통근시간 / 최대매매가 / 관심평형
        │
        ▼
[지오코딩 모듈]
  주소 → (lat, lng) 변환 (Kakao REST API)
  wp_id 발급 및 DB 저장
        │
        ▼
[반경 필터링]
  haversine 거리로 반경 R_km 이내 단지 선택
  (dual 모드: WP1 ∩ WP2 교집합)
        │
        ▼
[대중교통 경로 모듈]
  grid_key 기준 캐시 조회 (transit_cache)
  캐시 미스 → ODsay 멀티키 병렬 호출
  결과 저장 → transit_routes
        │
        ▼
[실거래 매칭]
  apartments JOIN transit_routes JOIN trade_recent
  평형 / 가격 / 통근시간 조건 필터
        │
        ▼
[추천 매트릭스 엔진]
  버킷 생성 (10분 단위 슬라이스)
  (bucket, pyeong) 슬롯별 최저가 1건 추천
  pick_reason 자동 생성
        │
        ▼
[LLM 코멘트 엔진] (비동기 BackgroundTask)
  (wp_id, apt_seq, pyeong_type) 캐시 조회
  캐시 미스 → Sonnet(추천) / Haiku(일반) 호출
  결과 DB 저장
        │
        ▼
[응답]
  cards + stats + buckets + llm_pending
```

## 6. 데이터 모델 핵심 테이블

| 테이블 | 역할 | 특허 관련성 |
|--------|------|------------|
| `workplaces` | 직장 주소 + 좌표 + wp_id | 독립항 1, 2 |
| `apartments` | 단지 마스터 + grid_key | 독립항 1 |
| `transit_cache` | (wp_id, grid_key) 단위 ODsay 캐시 | 종속항 4-D |
| `transit_routes` | 단지별 경로 rank 1~N | 독립항 1 |
| `trade_recent` | 최근 실거래 (평형·가격·층) | 독립항 1 |
| `apt_pt_friend_comment` | (apt_seq, pyeong_type, wp_id) LLM 캐시 | 종속항 4-C |
| `apt_walking_poi` | 도보권 시설 (POI 카테고리·거리) | 종속항 4-E |
