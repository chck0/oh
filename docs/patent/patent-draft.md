## 【명세서】

---

### 【발명의 명칭】

대중교통 통근시간 및 직장 위치 기반 부동산 자동 추천 시스템 및 방법

---

### 【기술분야】

본 발명은 정보 처리 및 부동산 정보 서비스 기술 분야에 관한 것으로, 구체적으로는 사용자의 직장 주소를 입력받아 대중교통 실측 통근시간과 부동산 실거래 데이터를 결합하여 최적의 매물을 자동 추천하는 시스템 및 방법에 관한 것이다.

---

### 【배경기술】

종래의 부동산 정보 서비스는 지역명, 가격대, 면적 등의 조건을 사용자가 직접 입력하여 검색하는 방식을 채택하고 있다. 그러나 이러한 방식은 직장까지의 실제 대중교통 이동 시간을 고려하지 않으므로, 사용자는 검색 결과를 얻은 후 별도의 지도 서비스를 이용하여 통근 가능 여부를 일일이 확인해야 하는 불편함이 있다.

또한 동일한 통근시간대라도 매물의 가격과 면적(평형)이 다양하게 분포하여, 사용자가 여러 매물을 직접 비교하여 최적 조합을 선택하는 데 상당한 시간과 노력이 필요하다. 특히 맞벌이 가구의 경우 두 직장 모두를 고려한 최적 거주지 선정은 더욱 복잡하며, 이를 지원하는 서비스는 현재 존재하지 않는다.

종래 기술의 문제점을 구체적으로 살펴보면:

(1) 부동산 플랫폼(네이버부동산, 호갱노노, 직방 등)은 지역 기반 검색을 제공하나, 사용자 직장으로부터의 대중교통 실측 통근시간을 검색 기준으로 활용하지 않는다.

(2) 지하철 노선도 기반 거리 추정 방식은 환승·대기 시간이 반영되지 않아 실제 통근시간과 괴리가 발생한다.

(3) 가격과 통근시간이라는 두 축을 동시에 고려하는 매트릭스형 추천 알고리즘이 부재하다.

(4) 두 명 이상의 직장 위치를 동시에 만족하는 매물을 자동으로 추천하는 서비스가 존재하지 않는다.

(5) 많은 수의 단지에 대한 대중교통 경로를 실시간으로 조회하면 응답 지연이 발생하며, 이를 효율적으로 캐싱·병렬 처리하는 기술이 필요하다.

본 발명은 상기한 종래 기술의 문제점을 해결하기 위해 안출된 것이다.

### 【발명의 내용】

#### 【해결하려는 과제】

본 발명이 해결하려는 과제는 다음과 같다:

(1) 사용자의 직장 주소를 기준으로 실제 대중교통 통근시간을 자동 산출하여 적합한 반경 내 매물을 필터링하는 것.

(2) 통근시간을 복수의 구간(버킷)으로 분류하고 매물의 평형 유형과의 교차 슬롯에서 최저가 매물을 자동 선정하는 매트릭스 추천 알고리즘을 제공하는 것.

(3) 두 명의 직장 위치를 동시에 고려하여 두 직장 모두 합리적 통근시간 내에 위치하는 매물만을 자동 추천하는 것.

(4) 대중교통 경로 API를 다수의 인증 키로 병렬 호출하고 지리적 격자 단위 캐싱을 통해 응답 지연을 최소화하는 것.

(5) AI 언어모델을 활용한 매물 설명 코멘트를 비동기 생성하고 캐싱하여 API 응답 지연 없이 사용자에게 제공하는 것.

#### 【과제의 해결 수단】

상기 과제를 해결하기 위하여 본 발명은 다음의 구성을 포함한다.

**제1 해결 수단: 통근버킷-평형 매트릭스 추천 방법**

직장 주소를 지오코딩하여 위도·경도를 획득하고, 대중교통 경로 API로 각 단지까지의 실측 통근시간(total_time_min)을 산출한다. 산출된 통근시간을 일정 구간(버킷)으로 분류하고, 각 버킷과 평형 유형의 교차 슬롯에서 실거래 최저가 매물 1건을 추천으로 선정한다.

**제2 해결 수단: 두 직장 교집합 필터링 방법**

제1 직장(WP1) 및 제2 직장(WP2)으로부터 각각의 반경 내에 동시에 위치하는 단지를 교집합으로 선정하고, 두 직장으로의 통근시간 중 최댓값 max(t1, t2)를 기준으로 버킷 분류 및 추천 매트릭스를 수행한다.

**제3 해결 수단: 대중교통 API 멀티키 병렬 호출 및 그리드 캐싱**

아파트 단지를 지리적 격자(그리드) 단위로 분류하고, 직장ID+격자셀 조합으로 캐시를 관리한다. 캐시 미스 셀에 대해 복수의 API 인증 키를 병렬 사용하여 호출 처리량을 극대화한다.

**제4 해결 수단: LLM 코멘트 비동기 생성 및 3키 캐싱**

(직장ID, 단지ID, 평형유형) 복합 키로 AI 생성 코멘트를 캐싱하고, 캐시 미스 시 BackgroundTask로 비동기 생성하여 API 응답 지연을 제거한다.

#### 【발명의 효과】

본 발명에 의하면 다음의 효과를 달성할 수 있다:

(1) 사용자가 직장 주소만 입력하면 해당 직장까지의 실제 대중교통 통근시간이 반영된 매물 목록을 자동으로 제공받을 수 있어 편의성이 크게 향상된다.

(2) 통근시간과 가격이라는 두 독립 축으로 구성된 매트릭스 구조로 각 (시간대, 평형) 조합에서 최적 매물이 자동 선정되므로, 사용자의 의사결정 부담이 현저히 감소한다.

(3) 맞벌이 가구가 두 직장 위치를 동시에 고려한 최적 거주지를 단일 검색으로 찾을 수 있어 기존 서비스 대비 차별적 가치를 제공한다.

(4) 지리적 격자 단위 캐싱과 멀티키 병렬 호출로 대량의 단지에 대한 경로 조회를 효율적으로 처리할 수 있다.

(5) AI 코멘트 비동기 처리로 사용자 체감 응답 속도를 개선하면서도 맞춤형 자연어 설명을 제공한다.

#### 제1 실시예: 통근버킷-평형 매트릭스 추천 방법

본 실시예에서는 단일 직장을 기준으로 하는 추천 방법을 설명한다.

**단계 1. 검색 조건 수신**

사용자 단말로부터 다음의 검색 조건을 수신한다:
- 직장 주소 (workplace_address): 도로명주소 또는 지번주소
- 최대 통근시간 (max_minutes): 10분~60분 정수, 기본값 60
- 관심 평형 유형 (pyeong_types): ['10평대', '20평대', '30평대', '40평대', '50평대+'] 중 복수 선택
- 최대 매매가 (max_price): 만원 단위 정수
- 최소 세대수 (min_kaptdaCnt): 선택사항

**단계 2. 직장 주소 지오코딩 및 직장 ID 발급**

외부 지오코딩 API(본 실시예에서는 Kakao REST API)를 호출하여 직장 주소를 위도(lat)와 경도(lng)로 변환한다. 변환 결과는 직장 테이블(workplaces)에 upsert하여 고유 식별자(wp_id)를 발급하고 반환한다. 이후 동일 주소 재검색 시 지오코딩 API를 재호출하지 않는다.

**단계 3. 유효 반경 계산 및 후보 단지 필터링**

최대 통근시간(T_max)에서 소정의 여유분을 차감한 유효 통근시간으로부터 소정의 계산식에 따라 지리적 반경(R_km)을 산출한다.

아파트 단지 DB에서 haversine 거리 공식으로 직장 좌표로부터 R_km 이내에 위치하고, is_apt=1 조건을 만족하는 단지를 후보로 선정한다.

**단계 4. 대중교통 경로 조회 및 캐싱**

각 후보 단지에 대해 다음의 순서로 통근 경로 데이터를 획득한다:

(4-1) 단지의 지리적 격자 키(grid_key)를 계산한다. 격자 키는 위도·경도를 일정 정밀도로 반올림한 복합 문자열로 구성된다.

(4-2) (wp_id, grid_key) 조합으로 캐시 테이블(transit_cache)을 조회한다. 캐시 히트 시 해당 결과를 사용하고 외부 API를 호출하지 않는다.

(4-3) 캐시 미스 셀에 대해 복수의 대중교통 경로 API 인증 키를 순환하며 병렬 호출한다. 복수의 인증 키를 동시에 활용하여 단위 시간당 처리 가능한 셀 수를 극대화한다.

(4-4) 특정 인증 키 호출 실패 시 다음 키로 자동 폴백한다.

(4-5) 응답된 경로 중 총 소요시간(total_time_min)이 effective_max_min 이하인 경우 통과(passed_filter=1)로 분류하고 transit_routes 테이블에 저장한다.

(4-6) 한 번의 검색 요청에서 처리 가능한 격자 셀 수는 서버 응답 제약에 따라 소정의 한도로 제한되며, 초과분은 미처리 상태임을 나타내는 메타 정보와 함께 반환하고 이후 검색에서 점진적으로 누적한다.

**단계 5. 실거래 매칭 및 카드 생성**

아파트 단지, 대중교통 경로, 실거래 테이블을 단일 쿼리로 조인하여 다음 조건을 만족하는 카드를 생성한다:
- transit_routes.rank = 1 (최적 경로 1건)
- trade_recent.pyeong_type IN (사용자 선택 평형 목록)
- trade_recent.deal_amount_int ≤ max_price

각 카드는 (단지명, 평형, 가격, 통근시간, 대중교통 요약) 정보를 포함한다.

**단계 6. 통근버킷-평형 매트릭스 추천 선정**

(6-1) 통근시간 버킷을 생성한다. 버킷은 소정의 기준 구간과 이후 소정 단위의 슬라이스로 구성되며, 최대 통근시간(T_max)에 따라 동적으로 결정된다.

(6-2) 각 카드에 total_time_min에 해당하는 bucket_idx와 bucket_label을 부여한다.

(6-3) (bucket_idx, pyeong_type) 조합을 슬롯으로 정의한다.

(6-4) 각 슬롯 내 카드를 price_low(최저 실거래가) 오름차순으로 정렬한다.

(6-5) 슬롯 내에서 이미 다른 슬롯에서 추천된 단지(apt_seq)를 제외하고, 최저가 카드 1건을 추천(is_recommended=True)으로 선정한다.

(6-6) 추천 카드에 추천 사유(pick_reason)를 자동 생성한다. 추천 사유는 슬롯명, 슬롯 내 매물 수, 동일 평형 최단버킷 대비 가격 차이를 포함한다. 예: "30~40분·30평대 중 최저가. 최단권보다 3,000만 저렴."

**단계 7. AI 코멘트 비동기 생성**

(7-1) 검색 API는 `llm_pending=true` 플래그와 함께 카드 목록을 즉시 반환한다. AI 코멘트 생성은 별도의 BackgroundTask로 분리하여 API 응답 지연을 방지한다.

(7-2) (wp_id, apt_seq, pyeong_type) 복합 키로 코멘트 캐시 테이블(apt_pt_friend_comment)을 조회한다. 캐시 히트 시 LLM을 호출하지 않는다.

(7-3) 추천 카드(is_recommended=True): 고성능 LLM을 호출하여 장점 1가지와 단점 1가지를 포함한 2문장 코멘트를 생성한다. 고성능 LLM 호출 실패 시 경량 LLM으로 폴백한다.

(7-4) 일반 카드: 경량 LLM을 호출하여 핵심 특징 단문 코멘트를 생성한다. 동시 호출 수는 소정의 한도로 제한한다.

(7-5) 클라이언트는 `llm_pending=true` 수신 후 폴링 엔드포인트(GET /api/comments)를 주기적으로 호출하여 코멘트 완료 여부를 확인한다.

#### 제2 실시예: 두 직장 교집합 기반 맞벌이 추천 방법

본 실시예는 제1 실시예를 기반으로 하며, 두 개의 직장 주소를 처리하는 추가 단계를 포함한다.

**단계 A. 제2 직장 주소 수신 및 유효성 검증**

제1 직장 주소(WP1)와 함께 제2 직장 주소(WP2)를 선택적으로 수신한다. WP2 입력 시 다음의 유효성 검증을 수행한다:
- WP1과 WP2의 주소 문자열이 동일한 경우 오류를 반환한다.
- WP2에 대한 지오코딩 결과의 wp_id가 WP1의 wp_id와 동일한 경우 오류를 반환한다.

**단계 B. 두 직장 반경 교집합 필터링**

WP1 좌표로부터 R_km 이내 AND WP2 좌표로부터 R_km 이내인 단지만을 후보로 선정한다:

```
후보 단지 = { apt | distance(apt, WP1) ≤ R_km AND distance(apt, WP2) ≤ R_km }
```

**단계 C. API 예산 동적 분배**

WP1과 WP2 각각에 대한 캐시 미스 셀 수(n1, n2)를 산출하고, 전체 API 호출 한도(TOTAL_LIMIT)를 다음의 규칙으로 분배한다:

```
half = TOTAL_LIMIT / 2
if n1 ≤ half and n2 ≤ half: take_1=n1, take_2=n2
elif n1 ≤ half: take_1=n1, take_2=min(n2, TOTAL_LIMIT-n1)
elif n2 ≤ half: take_2=n2, take_1=min(n1, TOTAL_LIMIT-n2)
else: take_1=half, take_2=half
```

WP1과 WP2의 대중교통 경로 조회를 비동기로 병렬 수행한다.

**단계 D. 대표 통근시간 산출 및 매트릭스 추천**

각 카드에 WP1→단지 통근시간(t1)과 WP2→단지 통근시간(t2)를 모두 부여하고, 대표 통근시간 = max(t1, t2)로 설정한다. 이후 제1 실시예의 단계 6과 동일한 버킷-평형 매트릭스 추천 알고리즘을 수행한다.

추천 사유는 두 직장 맥락을 포함한다: "두 직장 모두 30분 이내 · '20평대' 슬롯 최소가 (N곳 중)"

**단계 E. 하위 호환성 보장**

WP2 미입력 시 제1 실시예와 완전히 동일하게 동작한다. 두 모드 간 API 인터페이스, DB 스키마 변경 없이 단일 코드베이스로 처리한다.
