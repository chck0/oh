# Why Tree — BADUGI 핵심 결정 근거

핵심 설계 결정마다 "왜?"를 3단계로 파고든 문서.

---

## 1. 왜 직장 주소를 입력 기준으로 삼는가?

**Why 1**: 아파트 검색의 진짜 문제는 "어디서 봐야 할지 모른다"는 것이다.
**Why 2**: 사람들이 아파트를 고를 때 가장 먼저 하는 질문은 "거기서 출퇴근 가능해?"다.
**Why 3**: 통근시간은 삶의 질과 직결되고, 예산 다음으로 의사결정에 가장 큰 영향을 미친다.

→ **결론**: 검색의 시작점은 집이 아닌 직장이어야 한다.

---

## 2. 왜 통근버킷 × 평형 매트릭스로 추천하는가?

**Why 1**: "가장 좋은 집"이 아니라 "내 조건에서 가장 합리적인 집"을 찾아야 한다.
**Why 2**: 통근시간과 평형은 서로 다른 독립 축이다 — 30분대 20평대 vs 30분대 30평대는 다른 선택이다.
**Why 3**: 각 슬롯에서 최저가 1개만 뽑으면 비교 기준이 명확해진다.

→ **결론**: (버킷, 평형) 매트릭스의 각 슬롯에서 최저가 1개를 추천 카드로 제시한다.

---

## 3. 왜 ODsay를 쓰는가? (카카오 대중교통 API 대신)

**Why 1**: 대중교통 경로 데이터는 정확도가 서비스 신뢰의 핵심이다.
**Why 2**: ODsay는 셀 단위 호출이 가능하고, 다중 API 키로 병렬화가 용이하다.
**Why 3**: 카카오 대중교통 API는 요청당 비용과 rate limit이 상업적 규모에서 제약이 크다.

→ **결론**: ODsay 멀티키(최대 20개) 병렬 호출로 속도와 비용을 동시에 잡는다.

---

## 4. 왜 LLM 코멘트를 쓰는가?

**Why 1**: 수치만 나열하면 사용자는 비교를 포기한다.
**Why 2**: "친구가 카톡으로 한마디"하는 톤이 신뢰감과 접근성을 높인다.
**Why 3**: 점수·가중치는 룰로 산출하므로, LLM은 자연어 표현에만 개입 — 할루시네이션 리스크 최소화.

→ **결론**: Claude Sonnet(추천 카드 2~3문장)과 Haiku(일반 카드 한 줄)를 역할 분리하여 사용한다.

---

## 5. 왜 SQLite ↔ Supabase 이중 어댑터인가?

**Why 1**: 로컬 개발에서 Postgres를 띄우는 것은 진입 장벽이다.
**Why 2**: Vercel 프로덕션은 파일시스템이 read-only이므로 SQLite 사용 불가.
**Why 3**: `DATABASE_URL` 유무 하나로 자동 전환하면 코드 분기 없이 두 환경을 지원할 수 있다.

→ **결론**: `db.py`에서 환경변수를 감지해 커넥션과 SQL 방언을 자동 전환한다.

---

## 6. 왜 Vercel + Supabase인가?

**Why 1**: 초기 단계에서 인프라 운영 비용(시간·돈)을 최소화해야 한다.
**Why 2**: Vercel은 배포 파이프라인이 없고, Supabase는 Postgres를 관리형으로 제공한다.
**Why 3**: 두 서비스 모두 Free tier에서 MVP 검증이 가능하고, 트래픽 증가 시 유료로 전환 가능하다.

→ **결론**: Vercel(서버리스 함수 60초) + Supabase(pgBouncer Transaction mode)로 제로 운영 목표.

---

## 7. 왜 Vanilla JS인가? (React·Vue 대신)

**Why 1**: 현재 팀 규모와 MVP 단계에서 번들러·빌드 파이프라인은 오버헤드다.
**Why 2**: Vercel의 `outputDirectory: web`으로 정적 파일을 CDN에서 직접 서빙하면 빌드 단계가 없다.
**Why 3**: Kakao Maps SDK는 전역 스크립트 방식이라 Vanilla JS와 궁합이 자연스럽다.

→ **결론**: HTML + Vanilla JS + CSS로 빠르게 반복하고, 규모가 커지면 프레임워크를 도입한다.
