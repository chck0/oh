## Top Why

> **"사용자가 5분 안에 자신에게 맞는 아파트를 신뢰하고 결정할 수 있다"**

이것이 BADUGI의 궁극적 목적이다.
"5분"은 정보 과잉을 차단하겠다는 의지이고,
"신뢰"는 근거 없는 추천을 거부하겠다는 원칙이며,
"내게 맞는"은 개인화 없이는 의미 없다는 선언이다.

| 구현된 방법 | 설명 | spec |
|-----------|------|------|

| ✅ 통근버킷 × 평형 매트릭스 | (버킷, 평형) 슬롯별 최저가 1개 추천 | spec-03 |
| ✅ 가격 범위 필터 | min/max 예산 직접 입력 | spec-09 |

| ✅ 통근 경제성 표시 | 월 교통비·연간 통근 시간 가시화 | spec-28 |

## Middle Why 2 — 신뢰할 수 있는 데이터

**Why Up**: 추천이 아무리 개인화돼도 데이터가 잘못되면 결정을 망친다. 신뢰는 BADUGI의 생명선이다.

**How Down** (어떻게 데이터 신뢰를 확보하는가?):

| 구현된 방법 | 설명 | spec |
|-----------|------|------|
| ✅ ODsay 멀티키 병렬 호출 | 정확한 대중교통 경로, DB 캐시 | spec-02 |
| ✅ Why-price 태그 | "왜 싼지" 근거 자동 태깅 | spec-06, 11 |
| ✅ 공공데이터 원칙 | 국토부·한국부동산원 실거래만 사용 | manifesto |
| ✅ 웹 검색 출처 명시 | 학군·호재는 DuckDuckGo + URL 표기 | spec-24 |
| ✅ ODsay 키 자동 감시 | 죽은 키 감지 → 리포트 | Level 1 |
| ❌ 실거래 데이터 자동 갱신 | 국토부 API 주기적 갱신 파이프라인 없음 | **spec-29 후보** |

**대안 탐색 (How Down에서 떠오른 것)**:
- 카카오 대중교통 API → rate limit·비용 문제로 ODsay 선택
- 직방·호갱노노 데이터 → 공공데이터 원칙에 위배

**Why Up 연결**: 신뢰할 수 있는 데이터가 있어야 → 사용자가 결정을 "신뢰"할 수 있음

## Middle Why 3 — 이해하기 쉬운 설명

**Why Up**: 데이터가 있어도 해석이 어려우면 사용자는 결정을 포기하고 중개사에게 맡긴다. 친구가 "카톡으로 한마디" 해주는 톤이 필요하다.

**How Down** (어떻게 이해를 돕는가?):

| 구현된 방법 | 설명 | spec |
|-----------|------|------|
| ✅ Claude 코멘트 | Sonnet(추천 카드 2~3문장) + Haiku(일반 한 줄) | spec-05 |
| ✅ 친구 채팅 패널 | 단지 컨텍스트 자동 주입 + 깊은 질문 | spec-22~25 |
| ✅ 파일·이미지 첨부 분석 | 분양 팸플릿·계약서 PDF 직접 분석 | spec-27 |
| ✅ 경로 상세 팝업 | 통근시간 칩 클릭 → step-by-step 경로 | spec-20 |
| ❌ 스트리밍 채팅 응답 | 10초 이상 대기 이탈 방지 | 미정 |

**대안 탐색 (How Down에서 떠오른 것)**:
- 점수·등급제 → "비교가 판단을 만든다" 원칙에 위배, 배제
- 외부 채팅 앱 연동 → 단지 컨텍스트 유지 불가, 배제

**Why Up 연결**: 이해하기 쉬운 설명이 있어야 → "5분 안에 결정" 완성

## 수렴점 — 세 Middle Why가 만나는 곳

```
개인화된 후보    ─┐
                 ├──→  "5분 안에 신뢰하고 결정"
신뢰할 수 있는   ─┤         (Top Why)
데이터           │
                 │
이해하기 쉬운    ─┘
설명
```

세 축이 동시에 작동할 때만 Top Why가 실현된다.
하나라도 빠지면:
- 개인화 없음 → 내 상황과 무관한 추천
- 데이터 신뢰 없음 → 가격·경로를 못 믿음
- 설명 없음 → 수치 앞에서 포기

## 기존 설계 결정 로그 (11개)

> 아래는 개별 결정의 Why Up 상세 기록이다.
> 계층 구조에서 각 결정이 어느 Middle Why에 속하는지 표기했다.

### [개인화] 왜 직장 주소를 입력 기준으로 삼는가?
아파트 검색의 진짜 문제는 "어디서 봐야 할지 모른다"는 것이다.
사람들이 아파트를 고를 때 가장 먼저 하는 질문은 "거기서 출퇴근 가능해?"다.
통근시간은 삶의 질과 직결되고, 예산 다음으로 의사결정에 가장 큰 영향을 미친다.
→ **결론**: 검색의 시작점은 집이 아닌 직장이어야 한다.

### [개인화] 왜 통근버킷 × 평형 매트릭스로 추천하는가?
"가장 좋은 집"이 아니라 "내 조건에서 가장 합리적인 집"을 찾아야 한다.
통근시간과 평형은 서로 다른 독립 축이다.
각 슬롯에서 최저가 1개만 뽑으면 비교 기준이 명확해진다.
→ **결론**: (버킷, 평형) 매트릭스의 각 슬롯에서 최저가 1개를 추천 카드로 제시한다.

### [신뢰] 왜 ODsay를 쓰는가? (카카오 대신)
대중교통 경로 데이터는 정확도가 서비스 신뢰의 핵심이다.
ODsay는 셀 단위 호출이 가능하고, 다중 API 키로 병렬화가 용이하다.
카카오 대중교통 API는 요청당 비용과 rate limit이 상업적 규모에서 제약이 크다.
→ **결론**: ODsay 멀티키(최대 20개) 병렬 호출로 속도와 비용을 동시에 잡는다.

### [설명] 왜 LLM 코멘트를 쓰는가?
수치만 나열하면 사용자는 비교를 포기한다.
"친구가 카톡으로 한마디"하는 톤이 신뢰감과 접근성을 높인다.
점수·가중치는 룰로 산출하므로, LLM은 자연어 표현에만 개입 — 할루시네이션 리스크 최소화.
→ **결론**: Claude Sonnet(추천 카드 2~3문장)과 Haiku(일반 카드 한 줄)를 역할 분리하여 사용한다.

### [인프라] 왜 SQLite ↔ Supabase 이중 어댑터인가?
로컬 개발에서 Postgres를 띄우는 것은 진입 장벽이다.
Vercel 프로덕션은 파일시스템이 read-only이므로 SQLite 사용 불가.
`DATABASE_URL` 유무 하나로 자동 전환하면 코드 분기 없이 두 환경을 지원할 수 있다.
→ **결론**: `db.py`에서 환경변수를 감지해 커넥션과 SQL 방언을 자동 전환한다.

### [인프라] 왜 Vercel + Supabase인가?
초기 단계에서 인프라 운영 비용(시간·돈)을 최소화해야 한다.
Vercel은 배포 파이프라인이 없고, Supabase는 Postgres를 관리형으로 제공한다.
두 서비스 모두 Free tier에서 MVP 검증이 가능하고, 트래픽 증가 시 유료로 전환 가능하다.
→ **결론**: Vercel(서버리스 함수 60초) + Supabase(pgBouncer Transaction mode)로 제로 운영 목표.

### [인프라] 왜 Vanilla JS인가? (React·Vue 대신)
현재 팀 규모와 MVP 단계에서 번들러·빌드 파이프라인은 오버헤드다.
Vercel의 `outputDirectory: web`으로 정적 파일을 CDN에서 직접 서빙하면 빌드 단계가 없다.
Kakao Maps SDK는 전역 스크립트 방식이라 Vanilla JS와 궁합이 자연스럽다.
→ **결론**: HTML + Vanilla JS + CSS로 빠르게 반복하고, 규모가 커지면 프레임워크를 도입한다.

### [설명] 왜 친구 채팅을 별도 패널로 구현했는가? (spec-22)
추천 카드의 한 줄 코멘트만으로는 사용자가 깊은 질문을 할 방법이 없다.
외부 채팅 앱(카카오톡 등)으로 우회하면 맥락(단지 정보)이 사라진다.
채팅 패널을 인라인으로 두면 DB의 실거래·POI 데이터를 컨텍스트로 바로 주입할 수 있다.
→ **결론**: 단지별 컨텍스트(시세·실거래·POI)를 자동 주입한 채팅 패널을 화면 오른쪽에 슬라이드인으로 구현한다.

### [신뢰] 왜 채팅에 웹 검색 tool_use를 추가했는가? (spec-24)
DB에 없는 정보(학군·개발호재·정부정책)에 대한 질문이 실제로 많다.
웹 검색 없이 답하면 Claude가 "모른다"만 반복하거나 근거 없는 정보를 생성한다.
DuckDuckGo Instant Answer API는 API 키 없이 호출 가능해 추가 비용이 없다.
→ **결론**: tool_use agentic loop(최대 3턴)으로 검색 후 답변, 출처 URL 필수 표기.

### [신뢰] 왜 Word/PPT를 제외하고 이미지·PDF만 지원하는가? (spec-27)
python-docx·python-pptx는 lxml(C 확장 라이브러리)에 의존한다.
Vercel의 Python 서버리스 런타임은 C 확장 빌드 환경이 제한적이라 lxml 설치 실패.
부동산 문맥에서 가장 유용한 첨부는 분양 사진(이미지)과 계약서·분석 자료(PDF)다.
→ **결론**: 이미지(Claude Vision) + PDF(pypdf)만 지원. Word/PPT는 "PDF로 변환 후 첨부" 안내.

### [개인화] 왜 직접 검색 결과를 resultData.cards에 임시 추가하는가? (spec-26)
`showDetail(aptSeq)` 함수가 lat/lng를 `resultData.cards`에서 조회해 지도를 이동한다.
직접 검색한 아파트는 검색 결과에 없으므로 `cards`에서 찾지 못해 지도 이동이 안 된다.
`showDetail` 내부를 수정하면 기존 flow가 깨질 위험이 있다.
→ **결론**: `_openLookupDetail()`에서 결과 카드를 `resultData.cards`에 `_isLookup: true` 플래그와 함께 임시 삽입해 기존 함수를 재사용한다.
