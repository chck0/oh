# Spec 36: 대출 가능액 역산 계산기 (Affordability Calculator)

> 상태: Draft | 작성일: 2026-06-13 | 브랜치: claude/loan-affordability

## 1. Why (왜)

- **문제:** 현재 검색의 "가격 범위 → 최대"는 사용자가 **매수가능액을 숫자로 직접** 입력해야 한다. 그런데 첫 집 구매자가 실제로 아는 건 "내 현금 얼마 + 소득 얼마"지, "그래서 몇 억짜리까지 살 수 있나"가 아니다. 이 변환을 머릿속으로/외부 계산기로 하느라 검색 진입이 막힌다.
- **사용자가 얻는 것:** 보유 현금·연소득(+금리)을 넣으면 **"대략 N억까지"**를 즉석에서 역산해, 최대 가격 칸을 자동으로 채워준다. "얼마짜리를 봐야 하지?"의 1차 답.
- **데이터/비용:** 순수 산수. 새 API·새 DB·LLM 호출 0.

### 매니페스토 정합성 (중요)
- AGENTS.md 타겟은 "예산 범위를 **대략은 아는** 30대 1인 가구"이고, "예산 모르는 완전 초보 단계"는 의도적 제외다. → 본 기능은 **예산을 정밀 산정하는 사전심사 플로우가 아니라**, 대략 아는 사람이 숫자 칸을 빠르게 채우도록 돕는 **선택적 보조 위젯**으로 한정한다.
- "데이터 없으면 단정하지 않는다" → LTV/DSR/금리는 정책·개인 신용에 따라 달라지므로, 결과에 **"추정치 · 실제 한도는 은행 확인 필요"**를 항상 병기한다. 단정 표현 금지.

## 2. Scope (범위)

- **포함:**
  - search.html ② 가격 범위 안에 접이식 "💡 얼마까지 살 수 있나 계산" 보조 패널(기본 접힘)
  - 입력: 보유 현금(억/만원), 연소득(만원), (선택) 금리(%) · 대출기간(년)
  - 계산: LTV 한도 + DSR 한도 중 **작은 값**을 매수가능액으로 역산
  - "이 금액으로 최대 가격 채우기" 버튼 → 기존 `price-eok`/`price-man`에 주입
  - 추정치 + "은행 확인 필요" 안내 문구 상시 노출
- **제외(안 함):**
  - 백엔드·검색 API 변경 (오롯이 프론트 위젯, 기존 `max_price` 흐름 재사용)
  - 실시간 금리/우대금리/신용등급 연동, 보금자리·디딤돌 등 정책상품 자동판별
  - 자동 검색 실행 (값만 채우고 검색은 사용자가 누름)
  - 결과 화면 표시 (검색 입력 단계 한정)

## 3. 설계 (어떻게)

- **건드리는 파일:** `web/search.html` 만 (HTML 패널 + JS 계산 함수 + CSS). 백엔드 무관.
- **DB 변경:** 없음
- **API 변경:** 없음 (기존 `max_price` 파라미터 그대로 사용)

- **계산 로직 (역산):**
  ```
  입력: cash(만원), income(만원/년), rate(%, 기본 3.5), years(기본 30)
  ─ LTV 기준: 집값의 최대 LTV%까지 대출 → max_by_ltv = cash / (1 - LTV)
              (기본 LTV=70%. 즉 현금이 집값의 30% 이상이어야)
  ─ DSR 기준: 연 원리금상환 ≤ income × DSR%  (기본 DSR=40%)
              월상환여력 = income × DSR / 12
              대출가능액 = 원리금균등 역산(월상환여력, rate/12, years×12)
              max_by_dsr = cash + 대출가능액
  ─ 매수가능액 = min(max_by_ltv, max_by_dsr)  ← 둘 중 보수적인 값
  ─ 만원 단위 내림, 표시용 억/만원 분해
  ```
  - 상수(LTV 70 / DSR 40 / 금리 3.5 / 30년)는 JS 상단에 모아 한 곳에서 조정. 금리·기간은 사용자 입력으로 덮어쓸 수 있음.
  - 0/음수/공란 입력 → 계산 안 함(버튼 비활성 + 안내). cash만 있고 income 없으면 LTV 기준만, income만 있고 cash 0이면 DSR 기준만으로 추정하되 가정 명시.

- **주의 엣지케이스:**
  | 케이스 | 동작 |
  |---|---|
  | 현금·소득 둘 다 공란 | 버튼 비활성, "현금이나 소득 중 하나는 입력" |
  | rate=0 | 무이자 가정 → 단순 분할(원리금=원금/개월). 0 division 방지 |
  | 결과가 검색 상한(200억) 초과 | 상한으로 클램프 |
  | 결과 < 1억 | 그대로 표시 (작아도 정직하게) |
  | 모바일 | 기존 price-row 레이아웃 재사용, 접이식이라 화면 차지 최소 |

- **UX/톤:** 카톡 친구 톤 유지. 결과 예: "현금 1.5억 + 연소득 5천이면 **대략 4.8억까지** 볼 수 있어. (추정치라 실제 한도는 은행에서 확인해)" — 이모지 1개 이하, 단정 금지.

## 4. 완료 조건 (Acceptance Criteria)

- [ ] AC1: 가격 범위 섹션에 접이식 계산기 패널이 있고 기본 접힘 상태다
- [ ] AC2: 현금+연소득(+금리) 입력 후 매수가능액이 억/만원으로 표시된다 (LTV·DSR 중 작은 값)
- [ ] AC3: "최대 가격 채우기" 누르면 `price-eok`/`price-man`이 그 값으로 채워진다
- [ ] AC4: 결과에 "추정치 · 은행 확인 필요" 안내가 항상 보인다 (단정 표현 없음)
- [ ] AC5: 공란/0/이상치 입력 시 깨지지 않고 안내 메시지로 막는다
- [ ] AC6: 계산은 순수 프론트 — 백엔드/검색 API 변경 없음, 기존 검색 흐름 그대로 동작
- [ ] 모바일/데스크톱 레이아웃 정상

## 5. Open Questions

- Q1: 기본 LTV/DSR/금리 상수값 (70/40/3.5%) 이대로? → 보수적 기본값으로 두고 사용자가 금리만 조정 가능하게 제안.
- Q2: 계산기 위치 — 가격 범위 섹션 내부(제안) vs 별도 섹션?
- Q3: 결과를 "최대 가격"에만 채울지, "최소~최대" 범위 제안까지 할지 → v1은 최대만.
