# Spec 39: POI 마커 위치 신뢰성 — 이름 검증 후 배치

> 상태: Implemented | 작성일: 2026-06-15 | 브랜치: hjkang83

## 1. Why (왜)

- **버그:** 상세 패널에서 "서울홍제초등학교"(도보 6분·429m)를 눌렀는데 지도 핀은 **서울인왕초등학교** 위치에 찍힘. 말풍선 이름은 "홍제초"인데 핀 좌표는 다른 학교 → **라벨과 위치 불일치**.
- **원인:** `apt_walking_poi` 테이블엔 POI 좌표가 없다(이름·거리·도보시간만 저장). 그래서 `showPoiMarker`가 Kakao Places에 **이름으로 재검색** 후 `sort: DISTANCE`로 정렬된 **최근접 결과 `data[0]`을 무검증으로** 찍는다(spec-30 L33 "전역 검색 후 가장 가까운 결과 선택"). 키워드 검색이 느슨해 "서울홍제초등학교" 질의에 더 가까운 다른 학교(인왕초)가 1순위로 와도 그대로 배치됨.
- **임팩트:** 사용자가 보는 시설과 지도 위치가 달라 **서비스 신뢰성에 직접 타격**.

## 2. Scope (범위)

- **포함:**
  - `showPoiMarker`: Kakao 결과 중 **이름이 일치하는** 시설을 골라 배치(최근접 무검증 폐기).
  - 거리 sanity 체크: 후보의 직선거리가 알려진 도보거리(`distM`)와 모순되면 제외(도보거리는 직선거리 이상이어야 자연스러움).
  - 신뢰 가능한 후보가 없으면 **틀린 핀을 찍지 않고** 가벼운 안내(모르면 모른다 원칙).
- **제외(안 함):**
  - `apt_walking_poi` 스키마에 좌표 컬럼 추가/재적재(추후 근본 개선 후보로 기록).
  - `validateAndFilterPois`(존재 여부만 판정 → 오핀 위험 낮음)는 동일 헬퍼로 가볍게만 정합화.
  - 백엔드/DB/API 변경.

## 3. 설계 (어떻게)

- **건드리는 파일:** `web/result.html` 뿐.
- **DB/API 변경:** 없음

- **헬퍼 추가:**
  - `_haversine(lat1,lng1,lat2,lng2)` → 두 좌표 직선거리(m).
  - `_normName(s)` → 공백 제거 + 소문자(이름 비교용).
  - `_pickBestPoi(data, poiName, distM, aptLat, aptLng)` → 최적 후보 doc 또는 `null`.

- **`_pickBestPoi` 선택 규칙:**
  1. **정확 일치**(정규화 이름 동일) 후보 중 아파트에서 직선거리 최소인 곳 → 채택.
  2. 없으면 **부분 일치**(한쪽이 다른 쪽을 포함) 중 직선거리가 `distM*1.5+200m` 이내인 곳에서 최소거리 → 채택.
  3. 그래도 없으면 `null`(오핀 방지).

- **`showPoiMarker` 흐름:** 풀네임 검색 → `_pickBestPoi`. 없으면 `substring(0,6)` 폴백 검색 → `_pickBestPoi`. 둘 다 `null`이면 핀을 찍지 않고 `_poiNotice("지도에서 위치를 확인하지 못했습니다")` 표시(지도 이동/핀 없음).

- **엣지케이스:**
  | 케이스 | 동작 |
  |---|---|
  | 더 가까운 동명이종 시설 존재 | 이름 정확 일치 우선 → 올바른 시설 채택 |
  | Kakao에 "홍제초등학교"(시/도 접두 없음) 등록 | 부분 일치로 매칭 |
  | 진짜로 못 찾음 | 핀 미배치 + 안내(틀린 핀 금지) |
  | 좌표 어긋난 단지(spec-30 부수발견) | 거리 sanity로 엉뚱한 원거리 후보 제외 |

## 4. 완료 조건 (Acceptance Criteria)

- [x] AC1: 이름이 일치하는 시설에만 핀을 찍는다(최근접 무검증 폐기).
- [x] AC2: 직선거리가 도보거리와 모순되는 후보는 배제한다.
- [x] AC3: 신뢰 후보가 없으면 틀린 핀 대신 안내를 띄운다.
- [x] AC4: 시·도 접두 차이("서울○○초"↔"○○초") 등 부분 일치는 매칭된다.
- [x] 순수 프론트(result.html) — 백엔드/DB/API 무변경.
