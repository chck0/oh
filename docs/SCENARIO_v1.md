## 한 줄 정의

**점찍어둔 아파트 한 채에 대해 5명의 AI 분석가가 다관점 검증을 수행해, 사용자가 인생에서 가장 큰 거래를 후회 없이 결정하도록 돕는 "2차 의견(Second Opinion)" 시스템.**

---

## 포지셔닝

- **부동산 의사결정의 "Second Opinion"**.
- 부동산 중개인은 거래 성사가 수익이라 좋은 말 위주로 한다. 일반인은 그게 맞는 말인지 판단하기 어렵다.
- AI 에이전트들이 **다각도로 검증·반박**해서 균형 잡힌 판단을 도와주는 도구.
- 사라는 건지 사지 말라는 건지는 결국 사용자가 정한다. 다만, **어제는 몰랐던 것들을 오늘 안다.**

> "AI는 결정을 대신하지 않는다. 사용자를 더 똑똑하게 만들어준다."

## 전체 사용자 여정 (7 Scenes)

```
1. 진입 → 2. 기본 인터뷰 → 3. 매물 주소 입력 → 4. 분석 진행
   → 5. 에이전트 토론 → 6. 종합 리포트 → 7. 후속 액션
```

### Scene 01 — 진입 (인트로)

- 화면 메시지: "당신이 점찍은 아파트, 사기 전에 **다섯 명의 전문가**에게 검토받아 보세요."
- CTA: `[시작하기 →]`
- 톤: 한 문장으로 가치 제안. 부담 없는 시작.

### Scene 02 — 기본 인터뷰 (5~6개 짧은 질문)

**Design Principle: 질문은 짧게, 답변은 부담 없게.**

| 항목 | 질문 형식 | 답변 형식 |
|---|---|---|
| 보유 자산 | 현금/자산 대략 어느 정도? | 대략적 범위 (예: 한 2억) |
| 대출 한도 | 최대 얼마까지 받을 의향? | **총액 O** (월 상환액 X) |
| 회사 위치 | 회사가 어디? + 출퇴근 수단? | 구체적인 주소 + 지하철/버스/자가용 |
| 출퇴근 수단 | (위와 동시에) | (위와 동시에) |
| 우선순위 | 이 집에서 가장 중요한 건? | 1~2개만 꼽기 |

**대화 예시**:
- Q: 현재 보유하신 현금이나 자산은 대략 어느 정도이신가요?
- A: 한 2억 정도요.
- Q: 대출은 최대 얼마까지 받을 의향이 있으세요?
- A: 한 3.3억 정도요. 총 5.3억쯤 되겠네요.
- Q: 회사 위치는 어디세요? 출퇴근 수단도 알려주세요.
- A: 광화문 OO빌딩, 주로 지하철이요.
- Q: 이 집에서 가장 중요하게 보는 건 뭔가요? 한두 개만 꼽아주세요.
- A: 자산 가치 + 출퇴근 편의성이요.

### Scene 03 — 매물 주소 입력 ("여기, 바로 이 집")

- 입력 예: `광명시 OO아파트 105동 1203호`
- **Step 1 — 주소 변환**: 행정동/도로명 → API용 표준 주소
- **Step 2 — 지도 표시**: 위치에 핀 표시
- **Step 3 — 이 매물이 맞나요?**: 오타/오인 방지 컨펌

### Scene 04~05 — 5명의 전문가 분석 + 토론

> "5명의 전문가가 이 매물을 분석하고 있습니다. 약 30초 소요됩니다."

| # | 분석가 | 키 (코드용) | 영역 |
|---|---|---|---|
| 1 | **시세 분석가** | `market_analyst` | 실거래·호가·인근 시세 |
| 2 | **입지 분석가** | `location_analyst` | 역세권·학군·생활 인프라 |
| 3 | **리스크 분석가** | `risk_analyst` | 단지·거시 리스크 |
| 4 | **재무 분석가** | `finance_analyst` | 대출 한도·월 상환액 |
| 5 | **미래가치 분석가** | `future_analyst` | 개발 호재·5~10년 전망 |

각 분석가는 **자기 영역의 검증·반박**만 한다. 영역 침범 금지.

**토론 방식**: 찬반 / 레드팀 / 그 외 — **방식 미정 (논의 필요)**.

### Scene 06 — 종합 리포트 ("이게 진짜 내가 보고 싶었던 거다")

**한 화면 1페이지 요약** (가안 예시):

```
🏠 광명시 OO아파트 105동 1203호

종합 평점  ★★★☆☆  (3.2 / 5)
"입지·예산 적합도는 양호하나, 미래가치는 신중 검토 필요"

────────────────────────────────────────────────
💰 시세       ★★★★☆   오를 가능성 ↓
🏢 입지       ★★★★★   최상급
⚠️ 리스크    ★★☆☆☆   주의 필요
💳 재무       ★★☆☆☆   감당 가능 수준
🎯 미래가치   ★★★☆☆   호재 검증 필요
────────────────────────────────────────────────

5명 중 3명이 '신중 검토'를 권합니다.
핵심 쟁점: 입지·생활 만족도 vs. 5~10년 미래가치
```

### Scene 07 — 후속 액션 ("더 알아봐야겠다") — 3가지 가안

| 가안 | 설명 | 세부 |
|---|---|---|
| **A. 드릴다운** | 카드 클릭 시 세부 화면 진입 | 재건축 추진 이력 타임라인 / 관련 뉴스 기사 링크 / 동일 지역 유사 사례 비교 |
| **B. 에이전트와 대화** | 화면에 채팅창 | 사용자 직접 질문 / 여러 에이전트가 다른 관점으로 응답 / 에이전트끼리 추가 토론 |
| **C. PDF 저장·공유** | 가족·지인과 함께 결정 | 한 페이지로 저장 / 외부 공유 가능 / 후일 비교 자료로 보관 |

세 가지 모두 또는 조합 — **선택 가안**.

## 핵심 설계 방법론 — "아웃풋 이미지부터 역산"

```
1. 최종 아웃풋 이미지       → 한 화면에 보여줄 형태
2. 필요한 분석 결과         → 어떤 결론·지표가 필요한가
3. 필요한 에이전트 구성     → 관점·페르소나·오케스트레이션
4. 필요한 데이터·API        → 실존 여부 검증 필수
5. 인터뷰 질문 설계         → 분석에 진짜 필요한 것만
```

### 왜 역산하는가?

1. **데이터 못 가져오는 항목은 아웃풋에 담을 수 없다.**
2. **아웃풋 그릴 때부터 API 검증을 같이 해야 한다.**
3. **인터뷰 질문도 추상적이지 않게, 분석에 진짜 필요한 것만 묻는다.**

## 본 문서의 역할

- 컨셉·타겟·플로우·페르소나·아웃풋의 **단일 진실원**.
- 다른 모든 문서(README, MANIFESTO, WHYTREE, PREMORTEM, agents/*.md, glossary 등)는 이 문서와 정합해야 한다.

- 코드는 이 문서가 합의되고 다른 md가 정렬된 후에 구현한다 (B안 전면 피보팅).
