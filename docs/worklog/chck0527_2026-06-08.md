## 2026-06-08 22:30 — download_db.py pgBouncer 자동 회피 + hang 방지

**의도:** 새 환경에서 `python scripts/download_db.py` 실행 시 `.env`의 `DATABASE_URL`이 pgBouncer Transaction pooler(`:6543`)를 가리키면 서버사이드 named cursor가 silent hang(20분+) 됐음. 다음 사람이 같은 함정 안 빠지게 스크립트 자체에서 자동 회피.

**최종 상태:** `:6543` URL 그대로 두고 실행해도 자동으로 `:5432`(Session pooler)로 전환 + 사용자 알림. 670,317행 / 145.2 MB / 31초 완료 확인.

**건드린 파일·함수:**
- `scripts/download_db.py` — `_pg_connect()` 한 곳에서 5가지 방어 처리:
  1. 정규식 `(pooler\.supabase\.com):6543\b` → `:5432` 자동 재작성 (Session mode 전환)
  2. `autocommit=False` (named cursor는 트랜잭션 블록 필요, psycopg 3.3+)
  3. `prepare_threshold=None` (pgBouncer prepared statement 차단 호환)
  4. `connect_timeout=30` (네트워크 hang 30초 cap)
  5. `options="-c statement_timeout=600000"` (단일 쿼리 10분 cap)

**결정 / 버린 대안:**
- ❌ `.env`에서 6543 → 5432 수동 안내만 추가: 다음 사람이 또 같은 실수 가능
- ❌ Direct connect(`db.[ref].supabase.co:5432`)로 변경: IPv6 only라 일부 환경 불가
- ✅ Pooler 호스트 동일 + 포트만 5432: IPv4 OK + Session mode + named cursor 동작

**DB 변경:** 없음 (스크립트만 수정)

**함정·깨진 것:**
- pgBouncer Transaction mode는 named cursor 사용 시 에러도 안 내고 무한 대기 → silent hang이라 디버깅 어려움. timeout 옵션으로 보호.
- `\b` word boundary 안 붙이면 `:65432` 같은 포트도 잘못 매칭됨 (테스트로 확인).
