# 생애 첫 주택 구매 자문 시스템 용어 사전

> Premortem 시나리오 7 ("동상이몽") 방지를 위한 공용 용어 정의.
> 팀 전원이 같은 단어를 같은 뜻으로 쓰도록 한다.

---

## 🧑‍🤝‍🧑 에이전트 (Agent)

**정의**: 각각 고유한 페르소나(역할/말투/금지행동)를 가진 LLM 인스턴스.

- **MC 인터뷰어**: 자연스러운 대화로 BuyerProfile 수집. Stage 1 진행자
- **중개사 (broker)**: 입지·학군·교통·매물 특성 전담
- **재무설계사 (financial)**: DSR·대출 한도·자기자본·총취득비용 전담
- **시장분석가 (analyst)**: 실거래 P50·공급 리스크·타이밍·호가 적정성 전담
- **비서실장 (clerk)**: 중립적 상담록 정리 + 넥스트 액션 도출
- 각 에이전트는 `agents/*.md`에 명세된 규칙을 따른다.
- 내부 코드 키: `"broker"`, `"financial"`, `"analyst"`, `"clerk"` (변경 금지)

---

## 👤 BuyerProfile

**정의**: MC 인터뷰어가 Stage 1에서 수집하는 첫 주택 구매자의 상황 프로필.

| 필드 | 설명 | 배점 |
|------|------|------|
| `commute_location` | 주요 출근지 (예: 여의도, 판교) | 30점 |
| `budget_manwon` | 총 구매 예산 (만 원 단위) | 30점 |
| `own_funds_manwon` | 자기자본 (만 원 단위) | - |
| `monthly_payment_manwon` | 월 상환 가능액 (만 원 단위) | - |
| `family_size` | 가족 구성원 수 | - |
| `preferred_area` | 선호 지역 (예: 마포구, 성동구) | - |
| `preferred_size_sqm` | 선호 전용면적 (㎡) | - |
| `preferred_type` | 선호 주택 유형 (아파트/빌라/오피스텔) | - |
| `move_in_months` | 입주 희망 시기 (개월 수) | - |
| `notes` | 기타 특이사항 | - |

- COMPLETE_THRESHOLD = 60점 달성 시 Stage 2 자문으로 진행

---

## 🎤 MC 인터뷰 (Stage 1)

**정의**: 3단계 플로우의 첫 번째 단계. MC 인터뷰어가 자연스러운 대화로 BuyerProfile을 수집하는 과정.

- 딱딱한 폼 입력이 아닌 커피숍 대화 느낌
- 질문은 하나씩, 압박 없이 진행
- COMPLETE_THRESHOLD = 60점 (commute_location 30점 + budget_manwon 30점)
- 완료 후 Stage 2(전문 자문)로 자동 전환

---

## 📊 출처 명시 (Source Citation)

**정의**: 시장분석가·재무설계사가 수치를 말할 때 반드시 데이터 근거를 밝히는 규칙.

- 국토부 출처 형식: `[출처: 국토교통부 실거래가 공개시스템, API: getRTMSDataSvcAptTrade, LAWD_CD: {code}, N=?건, 조회일: {date}]`
- 데이터가 없으면 "이 부분은 데이터가 부족합니다"로 답할 것
- **핵심 가치 1번** ("출처 있는 숫자만 말한다")의 실천 기준

---

## 📍 P50 실거래가

**정의**: 국토교통부 실거래가 공개시스템 기준 최근 3개월 아파트 거래가의 **중앙값(50th percentile)**.

- 호가(매도 희망가)가 아닌 실제 계약 완료된 가격의 중앙값
- 시장분석가가 "호가 적정성" 평가 시 기준값으로 사용
- 예: "마포구 공덕동 전용 59㎡ P50 = 7.8억 [출처: 국토교통부, N=12건]"
- 산출 모듈: `src/molit_api.py`

---

## 🔴 역할 경계 (Role Boundary)

**정의**: 각 에이전트가 자기 전문 영역에서만 발언하도록 강제하는 규칙.

| 발언 OK | 발언 금지 |
|---------|----------|
| 중개사 → 입지, 학군, 교통, 매물 | 중개사 → DSR 계산, 시장 타이밍 예측 |
| 재무설계사 → DSR, 대출, 총취득비용 | 재무설계사 → 동네 추천, 시장 전망 |
| 시장분석가 → P50, 공급리스크, 타이밍 | 시장분석가 → 대출 계산, 투자 적합성 자문 |

---

## 🗺 3단계 플로우 (3-Stage Flow)

**정의**: 시스템의 핵심 진행 구조.

1. **Stage 1 — MC 인터뷰**: MC 인터뷰어가 BuyerProfile 수집 (COMPLETE_THRESHOLD=60점 달성까지)
2. **Stage 2 — 전문 자문**: 중개사·재무설계사·시장분석가가 BuyerProfile 기반 맞춤 자문
3. **Stage 3 — 호가 적정성**: 실거래 P50 vs 관심 매물 호가 비교 → 최종 매입 판단

## ✅ 넥스트 액션 (Next Action)

**정의**: 상담 결과로 도출되는 **동사로 시작하는 구체 행동**. 담당자와 기한이 반드시 붙는다.

- ❌ "알아보기"
- ✅ "마포구 공덕 전용 59㎡ 매물 현장 방문 예약하기 (이번 주 내)"
- ✅ "은행 대출 사전승인 신청하기 (KB·신한 각 1곳, 다음 주까지)"

## 📝 상담록 (Consultation Record)

**정의**: 비서실장 에이전트가 **고정 템플릿**에 따라 자동 생성하는 Markdown 문서.

- 필수 섹션: 고객 프로필 요약 / 각 전문가 의견(1줄) / 결정사항 / 보류사항 / 넥스트 액션
- `agents/clerk.md`의 템플릿이 유일한 진실원(Source of Truth)
- 파일 위치: `meetings/{YYYY-MM-DD}-{slug}.md`

## 🎯 Gold Standard

**정의**: "이 질문이 들어오면 이 응답이 나와야 한다"는 **기준 응답**. 각 페르소나 명세서에 예시가 포함되어 있고, 프롬프트 튜닝의 검증 기준이 된다.

- 데모 시나리오: **마포구 첫 주택 구매 검토**
- Mock 데모(`demo_mock.py`)가 Gold Standard 기반 응답을 보여준다

---

## 🔴 반론 강제 (Forced Objection)

**정의**: 시장분석가 에이전트가 **모든 응답에 반드시 하방 리스크/반론/의문을 1개 이상 포함**하도록 강제하는 규칙.

- "괜찮아 보입니다"만 하면 = 시장분석가의 실패
- 세 에이전트가 전부 같은 결론이면 → "빠뜨린 리스크는?" 자동 추가 질문

## 📡 MOLIT API (`src/molit_api.py`)

**정의**: 국토교통부가 제공하는 아파트 실거래 가격 정보 공공 API 연동 모듈.

- API명: `getRTMSDataSvcAptTrade`
- 최근 3개월 아파트 매매 실거래 데이터 조회 + P50(중앙값) 산출
- 출처 자동 생성: `[LAWD_CD: {code}, N=?건, 조회일: {date}]`
- 접근: [공공데이터포털](https://www.data.go.kr/) API 키 발급
- API 키 없을 때 샘플 데이터로 자동 fallback
- 서울 주요 구 지원: 마포, 성동, 강남, 서초, 용산, 송파, 영등포, 강서 등

---

## 🕷 크롤러 (`src/crawler.py`)

**정의**: 호갱노노·네이버부동산에서 시세 정보를 보완 수집하는 웹 크롤러 모듈.

- 대상: 호갱노노, 네이버부동산
- `CrawlerBlockedError`: 크롤러 차단 시 발생. graceful fallback으로 MOLIT API 전환
- MOLIT API와 함께 사용하여 실거래가 검증의 이중 안전망 역할

## 🎭 Mock 데모 (`demo_mock.py`)

## 🏷 MVP 정의

**"마포구 첫 주택 구매 검토" 시나리오가 3단계 플로우로 end-to-end 작동하는 5분 데모.**

1. **Stage 1**: MC 인터뷰어 → BuyerProfile 수집 (commute_location + budget_manwon)
2. **Stage 2**: 중개사·재무설계사·시장분석가 맞춤 자문
3. **Stage 3**: 실거래 P50 vs 호가 비교 + 비서실장 상담록 자동 생성
